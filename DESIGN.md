# CosmosDB ADBC Driver — Design

_An Arrow-native ADBC driver for Azure Cosmos DB (NoSQL / Core SQL API), conceived as a
modern analog of the Microsoft Cosmos DB **ODBC** driver: same "query documents, get a
tabular result" job, but returning **Apache Arrow** with its richer type system, plus
**DataFusion**-powered local query augmentation for things Cosmos can't do server-side._

Status: design draft, 2026-06-30. No code yet.

---

## 1. Goals

1. **Run a Cosmos query and return the documents as Arrow.** Three output representations,
   selectable per query:
   - **JSON** — each document as a JSON string in an `arrow.json` extension-typed column
     (**the default**: fully released in arrow-rs, lossless, universally consumable).
   - **Variant** — each document as an Arrow **Variant** column
     (`parquet_variant_compute::json_to_variant`); experimental / feature-gated.
   - **Inferred struct** — sample a configurable number of documents, infer an Arrow
     `Struct` schema, and project documents into it (`arrow_json` infer + `Decoder`).
2. **Augment Cosmos's query surface with DataFusion.** Push down whatever Cosmos can do
   cross-partition (via the experimental engine); run everything else (joins across
   containers, cross-source ops, unpushable expressions) locally in DataFusion.
3. **Modern auth.** Entra ID / managed identity / service principal **and** account key
   from v1 — closing the ODBC driver's single biggest gap.
4. **Cross-platform, columnar, streaming.** Ship as a portable ADBC `cdylib` usable from
   Python / R / Go / Java / DuckDB / Polars, streaming `RecordBatch`es (vs the Windows-only,
   row-at-a-time ODBC driver).

### Decisions locked with the user (2026-06-30)

| Question | Decision |
|---|---|
| Query interaction model | **Per-statement dialect toggle**: `native` (Cosmos SQL passthrough) vs `datafusion` (ANSI SQL over containers-as-tables), both first-class. |
| Cosmos read backend | **Experimental `azure_data_cosmos_engine` as the preferred push target** (full cross-partition surface server-side), **DataFusion as local compensation** for what it can't push. |
| Authentication | **Entra ID / managed identity / service principal + account key**, from v1. |
| Default output representation | **JSON string column (`arrow.json`)**; Variant and inferred-struct are opt-in. |

## 2. Reference material (what we are standing on)

- **`adbc-drivers/datafusion`** (`adbc-driver-datafusion` v0.25, DF 53 / arrow 58 / adbc 0.23,
  edition 2024) — **the skeleton we fork.** It already gives us the four `adbc_core` traits on
  four structs, a single `Arc<SessionContext>` threaded Database→Connection→Statement, the
  async→sync `Runtime{Handle|Tokio}` + `block_on` bridge, the
  `SendableRecordBatchStream → Iterator<RecordBatchReader>` adapter, `information_schema`-based
  `GetObjects`, bulk-ingest, and the `ContextInit` closure extension seam. It deliberately has
  **no custom `TableProvider`** — that gap is exactly our work.
- **`adbc_core` / `adbc_ffi` 0.23** (`apache/arrow-adbc`, `rust/core/src/sync.rs`) — the
  traits: `Driver→Database→Connection→Statement`, each `Optionable`. `Statement::set_sql_query`
  + `execute() -> Box<dyn RecordBatchReader + Send>` is the query seam. `export_driver!` in
  `adbc_ffi` does the C FFI export; crate is `["lib","cdylib"]`.
- **`driverbase` 0.1** (`adbc-drivers/driverbase-rs`, git-only) — reused as-is for `ErrorHelper`,
  `InfoBuilder`/`InfoRegistry` (`get_info`), `GetObjectsImpl` (`get_objects` Arrow assembly),
  `BulkIngestState`.
- **`template-rs`** — `cargo-generate` scaffold producing a compiling no-op driver. Handy for
  the initial layout; we'll bump its pins (0.22/arrow 57.2 → 0.23/arrow 58).
- **mashup-rs federation** (`docs/federation-learnings.md`) — the DataFusion federation
  playbook. Our case is its **§9** ("remote supports *less*"): capability-gated subtree folding
  + `supports_filters_pushdown`, SQL generated by unparsing the plan + a custom dialect, remote
  I/O deferred to stream-poll time, transport isolated behind a neutral value enum, one shared
  multi-threaded runtime.
- **Cosmos cross-partition state** (`rust-cross-partition-queries.md`) — the public
  `azure_data_cosmos` SDK does cross-partition `SELECT/WHERE` only; the full surface (ORDER BY,
  aggregates, GROUP BY, DISTINCT, TOP/LIMIT) needs the **experimental engine**, the gateway
  merge, or a hand-rolled pipeline. We use the experimental engine.
- **Arrow output types** — `arrow.json` (released), Variant via `parquet-variant` /
  `parquet-variant-compute` (`json_to_variant`, experimental, `variant_experimental` feature),
  JSON schema inference via `arrow_json::reader::infer_json_schema_from_iterator` + `Decoder`.

## 3. Architecture

Three crates, layered so DataFusion/Arrow/ADBC types never leak into transport (mashup-rs §10).

```
┌ adbc-cosmos  (cdylib + lib) ───────────────────────────────────────────────┐
│  Driver/Database/Connection/Statement (adbc_core) · options · error ·        │
│  runtime bridge · stream→Reader adapter · export_driver! · metadata          │
│  Statement dispatch:  dialect == native  → cosmos-client directly            │
│                       dialect == datafusion → SessionContext (federation)    │
└───────────────┬───────────────────────────────────┬──────────────────────────┘
                │ (datafusion dialect)               │ (native dialect)
┌ cosmos-datafusion (lib) ──────────────────────────┐│
│  CosmosCatalog/Schema providers · CosmosTable-    ││
│  Provider (inferred schema, supports_filters_     ││
│  pushdown) · CosmosExec (ExecutionPlan, defers    ││
│  I/O to poll time) · capability model             ││
│  (remote_can_execute) · sql_gen (Unparser +       ││
│  Cosmos dialect) · schema inference · JSON/        ││
│  Variant/struct RecordBatch builders              ││
└───────────────┬───────────────────────────────────┘│
                │                                      │
┌ cosmos-client (lib) — neutral, no DF/Arrow/ADBC ────▼──────────────────────────┐
│  auth (azure_identity: DefaultAzureCredential + account key) ·                 │
│  transport (azure_data_cosmos: connect, list dbs/containers, feed ranges,      │
│    single-partition exec, continuation paging) ·                               │
│  engine (azure_data_cosmos_engine: pull-based cross-partition pipeline —        │
│    plan + ranges → run() → single-partition queries → provide_data() → loop) · │
│  execute_query(container, cosmos_sql, opts) -> Stream<serde_json::Value>        │
└────────────────────────────────────────────────────────────────────────────────┘
```

### 3.1 `cosmos-client` — transport (neutral)
- **No DataFusion / Arrow / ADBC types.** Inputs are a container ref + a **Cosmos SQL** string;
  outputs are `serde_json::Value` documents (async `Stream`), plus sync wrappers via the shared
  runtime.
- **Auth** (`auth.rs`): `azure_identity` `DefaultAzureCredential` (managed identity, service
  principal, `az login`) **and** account key / connection string. One `CosmosCredential` enum.
- **Transport** (`transport.rs`): built on the public `azure_data_cosmos` SDK — connect, list
  databases/containers (for metadata), `read_feed_ranges()`, single-partition execution with
  `FeedScope::range` + continuation tokens. This is the transport the engine drives.
- **Engine** (`engine.rs`): drives the experimental `azure_data_cosmos_engine` pull-based
  pipeline — obtain the query plan (gateway plan endpoint), build a `Pipeline` for the plan +
  partition-key ranges, loop `run()` → execute the single-partition queries it asks for via the
  transport → `provide_data()` → repeat, yielding merged documents. Handles split repair (410)
  and merges through the engine.
- **Fast path:** a simple single-container `SELECT/WHERE` with no cross-partition merge can skip
  the engine and fan out via the SDK directly.

### 3.2 `cosmos-datafusion` — the DataFusion bridge
- **`CosmosTableProvider`**: `schema()` returns the container's inferred Arrow schema (sampled at
  bind time, cached); `scan(projection, filters, limit)` builds a Cosmos SQL query from the
  projection + accepted filters + limit and returns `CosmosExec`; `supports_filters_pushdown()`
  returns `Exact`/`Inexact`/`Unsupported` per filter per the capability model.
- **`CosmosExec` (`ExecutionPlan`)**: holds the generated Cosmos SQL + container + output schema;
  **defers all Cosmos I/O to stream-poll time** (`RecordBatchStreamAdapter` over a lazy stream) so
  we never nest Tokio runtimes; pins target partitions to 1 if any host UDF is involved.
- **Capability model (`pushdown.rs`)** — `remote_can_execute(plan_node) -> bool`, the federation
  brain. Cosmos-via-engine **supports** (single container): projection, `WHERE`
  (comparisons/IN/string & math funcs/`IS_DEFINED`/`ARRAY_CONTAINS`…), `ORDER BY`, aggregates
  (COUNT/SUM/AVG/MIN/MAX), `GROUP BY`, `DISTINCT`, `TOP`, `OFFSET…LIMIT`. It **cannot**: join
  across containers, union across containers/sources, correlated cross-container subqueries,
  arbitrary host UDFs. Those stay in DataFusion. Fold the **maximal** pushable subtree per
  container into one Cosmos SQL round-trip (mashup-rs §8, `TopDown`, "fold else recurse"); a join's
  two inputs each fold, the join runs locally.
  > **⚠️ Empirically corrected (2026-07-01, live emulator + engine v0.5.0).** The list above was the
  > *aspiration*; the pinned engine and Cosmos itself are much narrower. What we actually measured
  > (details: `journal/research/cosmosdb/odbc_driver.md` §"Engine surface", and the
  > `cosmos-pushdown-surface-empirical` memory):
  > - **Engine `SUPPORTED_FEATURES` (v0.5.0)** = `OffsetAndLimit, OrderBy, MultipleOrderBy, Top,
  >   NonStreamingOrderBy, Aggregate, HybridSearch`. **`GroupBy`, `Distinct`, `NonValueAggregate`,
  >   `CompositeAggregate`, `MultipleAggregates` are NOT advertised → gateway rejects them (400).**
  >   Only a single `SELECT VALUE <agg>` works; `GROUP BY`, `DISTINCT`, projected/multi aggregates
  >   are unreachable via the engine.
  > - **Null/undefined semantics diverge from SQL**, so most engine-pushable ops are *not*
  >   row/order-equivalent to DataFusion: `MIN(c.s)`→`null` (vs SQL `min`="x"), `COUNT(c.s)` counts
  >   JSON-null, `ORDER BY` places null/undefined first-ASC/last-DESC (opposite DataFusion default),
  >   and `ORDER BY … LIMIT` returns a *different row set*. Cosmos SQL has no `NULLS FIRST/LAST`.
  > - **Net:** the only fold that is both engine-supported **and** Exact is `COUNT(*)` →
  >   `SELECT VALUE COUNT(1) FROM c [WHERE …]`. Everything else (GROUP BY / DISTINCT / general
  >   aggregates / ORDER BY) stays in DataFusion — the correct, reference-validated path.
  > - **`AVG` pushdown is safe** where the engine accepts it: cross-partition `AVG` is computed
  >   count-weighted correctly (verified skewed partitions → true average, not average-of-averages).
  > - **Multi-column `ORDER BY` should stay local unless a composite index is known to exist** — real
  >   Cosmos requires a composite index on the sort tuple or the pushed query fails (the emulator is
  >   lenient and does not enforce this). This mirrors the Microsoft ODBC driver, whose
  >   `EnableSortPassdownForMultipleColumns` defaults **off** for exactly this reason (while
  >   `EnablePassdownOfAvgAggrFunction` defaults **on**).
- **`sql_gen.rs`**: unparse the (optimized) `LogicalPlan` with DataFusion's `Unparser` + a **Cosmos
  SQL `CustomDialect`**, then AST-rewrite the Cosmos-specific bits (`FROM c` aliasing,
  `SELECT VALUE`, property paths, `TOP`/`OFFSET…LIMIT`, Cosmos built-in function names). Never
  string-concat SQL. This is the most Cosmos-specific and highest-risk component.
- **Schema/output builders (`schema.rs`)**: sample N docs → `infer_json_schema_from_iterator`;
  build `RecordBatch` in JSON (`arrow.json` field metadata), Variant (`json_to_variant`, feature
  `variant`), or inferred-struct (`ReaderBuilder`/`Decoder`) form. Missing fields → null;
  type-mapping and the conflicting/heterogeneous-field policy (default: fall back to `Variant`, not
  `String`) are specified in **§3.5**.
- **`catalog.rs`**: Cosmos database→ADBC catalog, container→table (schema `"public"`), registered
  as lazy providers so listing is cheap and inference happens on first schema access.

### 3.3 `adbc-cosmos` — the ADBC/FFI shell
- Forks the DataFusion driver's `lib.rs` skeleton: four traits, `Arc<Runtime>`,
  stream→`RecordBatchReader`, `export_driver!(AdbcDriverCosmosDbInit, CosmosDriver)`, `["lib",
  "cdylib"]`.
- **Statement dispatch** on the `adbc.cosmos.dialect` option:
  - `native` → send `set_sql_query` text (Cosmos SQL) to `cosmos-client` for the target container
    (option/`USE`-style), stream documents, build the `RecordBatch` in the configured output mode.
    No DataFusion. This is the ODBC-analog path.
  - `datafusion` → run the text as ANSI SQL through the `SessionContext` with Cosmos containers
    registered as `CosmosTableProvider`s; pushdown + local compensation as above.
- **Metadata**: `get_objects` / `get_table_schema` list databases/containers via `cosmos-client`
  (cheap) and infer schema on demand; `get_info` via `driverbase::InfoRegistry`.

### 3.5 Schema inference & type mapping

Inference only matters for the `struct` and (to a lesser degree) `variant` outputs; the default
`json` output is a lossless passthrough that needs none. Two layers:

- **Layer 1 — value → type.** Given sampled JSON values for a field, pick an Arrow type.
- **Layer 2 — representation.** Choose how to *carry* a field: a typed Arrow type, a `Variant`,
  or a JSON-string fallback. This is a lever the ODBC driver lacks (it has only a flat SQL type
  system), and it's where Arrow lets us beat it.

**Guiding principle: conservative, lossy-safe defaults; richer targets are opt-in.** This mirrors
the Microsoft ODBC driver (our analog), whose inference we measured live (see
`journal/research/cosmosdb/odbc_driver.md`): it infers `Int64` for integral numbers (big ints kept
exact), `Double` for any-fractional, `Boolean`, and otherwise `String` — and deliberately does
**not** guess dates, timestamps, decimals, or GUIDs, and widens type-conflicting fields to
`String`. It also *flattens* nested objects and normalizes arrays into child tables (both artifacts
of a flat type system we don't share).

**Base inference (`struct`):** arrow-json `infer_json_schema_from_iterator` over the sample →
`Int64` / `Float64` / `Boolean` / `Utf8` / **`Struct`** / **`List`** (real nesting, not the ODBC
flatten) / null-for-missing, everything nullable. This already matches the ODBC driver on scalars
and beats it on structure.

**Confirmed policy decisions (2026-07-01):**

1. **Numbers — one "numeric fidelity" knob, applied uniformly (`number_inference`).** Default
   `float64`: typed columns → `Float64`, variant numbers → `Double`. Opt-in `decimal` (with
   `decimal = p,s`): typed columns → `Decimal128(p,s)`. The knob governs *both* representations so
   we never ship a surprising typed-vs-variant asymmetry. Integers are always exact `Int64` (both
   representations agree). **Empirical caveat (verified against `parquet-variant-compute` 58.3):
   `json_to_variant` currently encodes *all* fractionals — and integers beyond `i64` — as `Double`;
   it does not yet emit Variant decimals** (upstream `// Todo`). So under `number_inference=decimal`
   the decimal typing applies to `struct` columns today; `variant` numbers stay `Double` until the
   library supports decimal encoding (a library limitation we document, not a designed asymmetry —
   revisit when upstream lands it, or hand-roll only if a user needs it sooner). Decimal typing
   **recurses** into nested `Struct`/`List` (any nested `Float64` → `Decimal128`).
2. **Heterogeneous / type-conflicting fields (`heterogeneous`).** These would otherwise crash —
   `infer_json_schema_from_iterator` **errors** on a path that's a scalar in one doc and an
   object/array in another, and even for scalar-vs-scalar the `Decoder` rejects the non-string
   values it inferred as `Utf8`. Fix: a **recursive shape pass** (shared
   `cosmos_datafusion::normalize`) merges a JSON shape across sampled docs and stringifies exactly
   the conflicting nodes **at any depth** to `Utf8` before infer/decode — homogeneous nested
   `Struct`/`List` typing is preserved. A **top-level** conflicting field can instead be carried as
   a self-describing Arrow Variant column via `json_to_variant` (**`variant`**, feature-gated);
   everything else (nested conflicts, and top-level in `string` mode) is stringified. `string` is
   the effective default because Variant is behind the opt-in `variant` feature. Robust across
   **all read paths**: `struct` output (`inference.rs`), metadata (`inference::infer_schema`), and
   the `datafusion` provider (`convert.rs`: shape-normalize at infer; schema-guided
   stringify-where-`Utf8` at decode, which also tolerates out-of-sample type drift). Only the §3.5
   *type transforms* (below) remain top-level; conflict normalization is fully recursive.
3. **Dates/datetimes from strings (`infer_temporal`, default `off`).** ISO-8601 string → `Date32`/
   `Timestamp` is pattern-guessing and can misfire on a field that merely looks date-like; the ODBC
   driver doesn't do it, so we default off and make it explicit opt-in. **Recurses** into nested
   `Struct`/`List` (a `Utf8` becomes temporal only when *every* sampled value at that path is ISO).
4. **Epoch numbers → timestamps (`epoch_fields`).** A bare number can't be distinguished from an ID
   or count, so Unix-timestamp inference is **never** automatic — the user names the fields and unit
   (`_ts:s`, `createdAt:ms`). Matches **top-level** field names only (nested paths aren't addressable).
5. **Nested objects/arrays → `Struct`/`List`** by default (lossless), with `Variant` or JSON-string
   as per-field fallbacks under heterogeneity or low sample confidence.

**Known precision limitation (track):** integers larger than `i64` (Cosmos can produce them) lose
precision — `Double` in `variant` today, and `arrow-json` would infer/overflow `Int64` in `struct`.
Rare, but a real edge (`_ts`/epoch/large IDs need care; ties into §4 numeric policy). Candidate
fix: promote out-of-`i64` integers to `Decimal128`/`Decimal256`.

## 4. Cross-cutting concerns
- **One shared multi-threaded Tokio runtime** (`Arc<Runtime>`, `Handle|Tokio`); `block_on` only at
  the ADBC boundary and for eager schema fetch; **never `block_on` on a runtime thread** — defer
  Cosmos I/O to stream-poll time inside `CosmosExec`.
- **Type/precision policy**: explicit, tested `serde_json::Value` → Arrow mapping. Decide numeric
  representation deliberately (keep i64/f64/`Decimal128` distinct rather than collapsing to f64;
  Cosmos numbers are JSON doubles but IDs/epochs need care). Heterogeneous fields → union or a
  JSON fallback column, never a silent lossy cast.
- **Diagnostics from day one** (mashup-rs §12): a plan/SQL trace switch printing every fold
  decision (folded / rejected: non-Cosmos leaf / rejected: unsupported op / rejected: has UDF) and
  the generated Cosmos SQL.
- **Version matrix** (align to the DataFusion driver to keep the arrow ABI consistent across FFI;
  all verified against cloned sources 2026-06-30):
  `arrow = 58`, `datafusion = 53`, `adbc_core`/`adbc_ffi = 0.23` (matches the skeleton; arrow-adbc
  main is already 0.24 with arrow pinned `>=58,<60`, so a later bump is low-risk),
  `parquet-variant-compute = 58` (feature-gated), `object_store = 0.13`, `tokio = 1`,
  `sqlparser` per DF 53, edition 2024, MSRV ≥ 1.85.
  - **`azure_data_cosmos = "=0.29.0"`** (crates.io, **exact pin**) — the last version exposing the
    `preview_query_engine` seam; SDK main (0.37) removed it. `azure_identity ≈ 0.30` (crates.io).
  - **`azure_data_cosmos_engine` = git dep pinned by rev** — NOT published to crates.io; workspace
    v0.5.0, default feature `query_engine` (pulls `azure_data_cosmos/preview_query_engine`).
  - The Cosmos transport stack is **REST/JSON and arrow-free**, so it cannot conflict with the
    arrow 58 / DF 53 layer — the layering is clean by construction.
- **Bonus, near-free**: because we inherit DataFusion's file/URL/object-store providers, the
  `datafusion` dialect can **join Cosmos containers against Parquet/CSV/S3** with no extra code.

## 5. Phased plan

### Phase 0 — Foundations & de-risking spikes
- Stand up the Cargo workspace; generate from `template-rs`, bump pins to arrow 58 / adbc 0.23 /
  DF 53 / edition 2024; produce a compiling no-op `cdylib` that a driver manager loads.
- **Spike A (transport+auth):** `azure_data_cosmos` as a git dep; connect with account key **and**
  Entra `DefaultAzureCredential`; run a cross-partition `SELECT/WHERE`; get `serde_json::Value`
  docs. _Validates build + auth + transport._
- **Spike B (engine — biggest unknown):** `azure_data_cosmos_engine` as a git dep; drive the
  pull-based pipeline for an `ORDER BY` / aggregate query end-to-end (incl. obtaining the query
  plan). _Confirms the experimental engine's Rust API is usable; resolves plan-fetch mechanism._
- **Spike C (output):** build a `RecordBatch` from `Vec<serde_json::Value>` in all three modes;
  round-trip through Python `adbc_driver_manager` / pyarrow to confirm `arrow.json` passes through
  and Variant feature-gating works.

### Phase 1 — Native-dialect MVP (the ODBC analog)
- `cosmos-client`: auth + transport + engine → `execute_query` streaming documents.
- `adbc-cosmos`: Database/Connection/Statement, options, `native` dialect path → `RecordBatch` in
  the configured output mode (JSON default). `get_objects`/`get_table_schema` listing + inference.
- **Ship:** a Python client runs Cosmos SQL and gets an Arrow `arrow.json` column back.

### Phase 2 — DataFusion federation (`datafusion` dialect)
- `cosmos-datafusion`: `CosmosTableProvider` + `CosmosExec` + `supports_filters_pushdown`, the
  capability model, `sql_gen` (Unparser + Cosmos dialect), catalog/schema providers.
- `adbc-cosmos`: `datafusion` dialect path — ANSI SQL over registered Cosmos tables; fold maximal
  pushable subtrees to Cosmos (engine), compensate locally (joins/cross-container/unpushable).
- Plan/SQL trace switch; connectionless SQL-generation tests. Free: Cosmos⋈Parquet/CSV.

### Phase 3 — Hardening & fidelity
- Full type mapping + numeric-precision policy; Variant to GA behind feature; struct inference
  across partitions with configurable N + confidence + heterogeneous-field handling; continuation
  robustness, split repair, 429 retries, consistency level, cancellation, `execute_partitions` for
  parallel reads; Python ADBC validation harness mirroring the DataFusion driver's `validation/`.

## 6. Open questions / risks
1. **Experimental engine API stability & plan fetch.** `azure_data_cosmos_engine` is
   experimental/unpublished; we must obtain the query plan (gateway plan endpoint vs native DLL).
   Spike B resolves feasibility; API churn is an ongoing risk (pin revs).
2. **Cosmos SQL dialect generation.** Unparsing DataFusion plans to Cosmos SQL (`SELECT VALUE`,
   `FROM c` aliasing, property paths, Cosmos built-in functions, `TOP`/`OFFSET…LIMIT`) is the
   hardest bespoke component. Start with a conservative supported set; expand behind tests.
2. **Unpublished Azure crates.** Both Cosmos crates are git-only; pin exact revs; expect churn.
4. **Output ↔ pushdown tension.** JSON/Variant single-column output gives DataFusion nothing to
   push except `id`/partition-key. The `datafusion` dialect benefits from `struct` inference; the
   `native`/JSON path is for raw retrieval. Document this coupling; consider promoting `id`,
   partition key, and `_ts` to real columns alongside a `document` JSON column.
5. **Schema inference vs schemaless reality.** Sampling misses out-of-sample fields and hits type
   conflicts; default to JSON, treat `struct` as opt-in for known-homogeneous containers, and
   provide a JSON fallback for non-conforming fields.
6. **Engine path anchors us to `azure_data_cosmos` 0.29** (a frozen SDK version); the seam is
   explicitly "unstable, may be removed" and *was* removed by 0.37. Mitigation: keep the transport
   behind the neutral `cosmos-client` surface so we can swap to (a) driving the engine's **native**
   `QueryPipeline` API directly over raw REST — plan via the gateway
   `x-ms-cosmos-is-query-plan-request: true` header, pk-ranges via `/pkranges`, both documented in
   the engine's `pipeline.rs` — or (b) the latest SDK with SELECT/WHERE-only + DataFusion doing the
   full merge, without touching the DataFusion/ADBC layers.

## 7. Verification findings (clone pass + Spike B, 2026-06-30)

- **Nested §3.5 type transforms — DONE, live-verified (2026-07-02).** `number_inference=decimal`
  (`Float64` → `Decimal128`) and `infer_temporal` (ISO string → `Date32`/`Timestamp`) now recurse
  into nested `Struct` fields and `List` elements — the top-level-only `decide_type`/`FieldProfile`

- **Nested/recursive conflict handling — DONE, live-verified (2026-07-02).** Extended
  heterogeneous handling to conflicts at *any depth*. New shared `cosmos_datafusion::normalize`:
  computes a merged JSON *shape* across sampled docs and stringifies exactly the conflicting nodes
  (nested object fields, array elements) to `Utf8` before infer/decode — previously these caused an
  infer error (scalar-vs-object) or decode error (non-string into inferred `Utf8`). Used by
  `struct` output, metadata, and the `datafusion` provider (which also does schema-guided
  stringify-where-`Utf8` at decode, tolerating out-of-sample drift). The old top-level-only
  exclusion logic (in `inference.rs` and `convert.rs`) is replaced by this. `mixed` seed container
  gained a nested-conflict field (`meta.v`). Tests: `normalize` unit tests, `inference` nested unit
  tests, live (struct nested `meta.v` → Utf8, datafusion on `mixed`), `roundtrip.py` 20/20.
- **Heterogeneous-field handling — DONE, live-verified (2026-07-02).** Type-conflicting fields
  (a scalar in one doc, an object/array in another) crashed inference/decode everywhere. Now
  detected up front and handled robustly across all read paths: `struct` output builds them as a
  stringified `Utf8` column (default) or, with the `variant` feature, a self-describing Arrow
  **Variant** column (`heterogeneous=variant`); metadata (`inference::infer_schema`) and the
  `datafusion` provider (`convert.rs`) exclude them from arrow-json inference and represent them as
  `Utf8` (the provider also stringifies `Utf8`-typed values at decode, tolerating out-of-sample
  drift). Seed gained a `mixed` container (conflicting `val`). Tests: unit (both feature sets),
  live (struct string + variant, datafusion on `mixed`), `validation/roundtrip.py` 20/20.
- **Per-container schema caching — DONE (2026-07-02).** The `datafusion` dialect previously
  re-sampled a container on every `CosmosSchemaProvider::table()` resolution (and rebuilt the

  SchemaRef>>`), owned by the `Connection` and threaded Connection → Statement →
  `register_cosmos_schema` → provider, so inferred schemas are memoized for the connection's

  `get_table_schema` and `get_objects` (column depth) go through `sample_schema`, which checks the
  cache before sampling — so a container's schema is inferred once per connection whether first
  touched by a query or a metadata call (best-effort snapshot; whichever path samples first wins;
  empty containers aren't cached). Unit test: `sample_schema` cache hit skips networking.

  - **Filter pushdown DONE, live-verified.** New `predicate.rs`: `supports_filters_pushdown`
    returns `Exact` only for a small, provably row-equivalent set — column-vs-literal comparisons
    (`= != < <= > >=`), `IN`/`NOT IN`, `IS NULL`/`IS NOT NULL`, and `AND`/`OR` of those; everything
    else → `Unsupported` (DataFusion keeps it). Accepted filters translate to a Cosmos `WHERE`
    clause appended by `build_scan_sql` before `OFFSET…LIMIT`. **Null/undefined/3VL trap:** each
    column comparison / `IN` leaf is wrapped `IS_DEFINED(c["x"]) AND NOT IS_NULL(c["x"]) AND (…)`
    so it is two-valued and matches DataFusion's "a null excludes the row" semantics exactly
    (a raw `c["x"] != 5` / `NOT IN` is *true* for a JSON `null`, which would corrupt results).
    `IS NULL` → `(NOT IS_DEFINED(x) OR IS_NULL(x))`; `IS NOT NULL` → the guard itself. Field-name
    bracket-quoting/escaping now shared via `predicate::cosmos_property` (escapes `\` then `"`).
    Connectionless unit tests assert both the generated SQL and the Exact/Unsupported decisions;
    a live `#[ignore]` test (`WHERE "mergeOrder" > 25` → exactly the 25-row subset) confirms the
    clause reaches Cosmos (DataFusion drops the `Exact` filter, so a wrong translation would return
    all 50).
